
Plan: remove the signup confirmation requirement and fix the auth flow

What’s wrong now
- The backend auth setup is still requiring email confirmation for email/password signups.
- The signup UI currently treats signup as fully successful even when the auth system may not return a live session.
- Any account created before the setting is changed may still be stuck in an unconfirmed state.

Implementation order
1. Update backend auth settings
   - Turn off email confirmation for email/password signups so new accounts are auto-confirmed immediately.
   - Leave password reset emails intact.

2. Fix the signup flow
   - Update `src/hooks/use-auth.ts` so signup handling checks the returned auth result, not just whether there was an error.
   - Update `src/pages/SignUp.tsx` so it only shows the “welcome” success path when the user/session is actually ready, and otherwise shows a clear fallback message.

3. Improve login error handling
   - Update `src/pages/Login.tsx` to show friendlier auth messages instead of raw backend errors, especially for any leftover legacy unconfirmed-account case.

4. Resolve existing affected account(s)
   - Check whether the current problem account was created while confirmation was still enabled.
   - If yes, fix that account state so the user can log in without relying on the broken confirmation email flow.

5. Verify end-to-end
   - A brand-new signup works without any confirmation email step.
   - The new user is signed in right away.
   - The currently affected user can log in.
   - Forgot-password/reset still works.

Technical details
- Files to update: `src/hooks/use-auth.ts`, `src/pages/SignUp.tsx`, `src/pages/Login.tsx`
- Backend change: authentication settings in Lovable Cloud
- No database schema, RLS, or autogenerated client changes are needed
